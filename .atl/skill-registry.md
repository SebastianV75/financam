## Compact Rules

Pre-digested rules per skill. Delegators copy matching blocks into sub-agent prompts as `## Project Standards (auto-resolved)`.

### go-testing
- Use table-driven tests for multiple test cases ( slices of structs with name/input/expected/wantErr)
- Use `t.Run()` to name subtests, not comments
- Golden file testing: read from `testdata/` directory, compare actual vs expected
- Bubbletea TUI: use teatest library for programmatic test interaction
- Mock interfaces, not concrete types — use fakes
- For async: use `assert.Check` pattern with goroutine-safe check

### skill-creator
- A skill needs a trigger (when AI should load it), not just a description
- SKILL.md structure: frontmatter + Critical Patterns + Rules
- Skills belong to user-level (global) or project-level (workspace)
- Allowed tools must be explicit in frontmatter
- Don't create skills for one-off tasks or trivial patterns

### judgment-day
- Launch TWO independent sub-agents simultaneously via `delegate` (never sequential)
- Each gets identical target, neither knows about the other (blind review)
- Synthesis: merge findings, apply fixes, re-judge until both pass OR escalate after 2 iterations
- Must resolve project standards before launching (get skill registry, inject rules)

### branch-pr
- Follow issue-first enforcement: requires issue exists BEFORE branch/PR
- Use `gh` CLI for all GitHub operations
- PR title: imperative mood, concise (50 chars max)
- PR body: Markdown with Summary section (1-3 bullet points)

### issue-creation
- Follow issue-first enforcement: no code change without prior issue
- Use `gh issue create` or GitHub web interface
- Include reproduction steps for bugs
- Labels: categorize appropriately (bug, feature, enhancement, documentation)
